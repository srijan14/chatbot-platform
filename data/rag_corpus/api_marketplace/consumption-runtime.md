# API Consumption Flow via Portal and Runtime

## Portal Consumption Flow

From a consumer's perspective, the portal flow for consuming APIs is:

1. Select the desired gateway (Intranet or Internet) and workspace.
2. Create an application if not already created.
3. Add APIs to the application.
4. Submit for approval and, once approved, obtain the necessary credentials
   (client ID and secret) generated by the system.

This ensures that every API call at runtime can be traced back to a specific
application configuration, including its IP whitelist and rate limits.

## Runtime Consumption Through KONG and OAuth

At runtime, APIs must be called via **KONG URLs**, not the original backend
endpoints. This enforces consistent cross-cutting concerns (security, rate
limiting, routing) and hides backend service details from consumers.

Required headers when calling APIs through KONG include:

- **Host** — set to the workspace value, which drives routing to the correct API
  cluster.
- **Authorization** — set to an OAuth access token.
- **x-api-key** — an optional header if additional key-based validation is
  configured.

The OAuth flow follows the **client credentials grant** pattern:

1. Generate an access token by making a POST request to `/oauth2/token` with
   `grant_type` set to `client_credentials` and including `client_id` and
   `client_secret`.
2. Call the target API using the obtained token in the Authorization header,
   along with the correct Host header for the workspace.

KONG then handles authentication, rate limiting, and routing decisions, acting
as the central enforcement point in front of all APIs.

## Special Considerations: Selfcare / SSO Consumption

The gateway feature in the Selfcare context is available only for **SSO (Single
Sign-On)** users and requires a dedicated user role. This means some
gateway-related capabilities are restricted to authenticated SSO users who have
appropriate authorization.

For legacy login paths, users follow the old, pre-gateway flow, which suggests
that the project may be in a transitional phase where both legacy and new
gateway-based consumption patterns coexist. Over time, the intent would
typically be to migrate all users to the SSO and gateway model for consistency
and improved security.
